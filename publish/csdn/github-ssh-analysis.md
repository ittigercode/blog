# GitHub SSH 深度研究报告：从密码嗅探到 Ed25519

> **摘要：** 用横纵分析法深度研究 GitHub SSH——纵向追溯 1995 年 Tatu Ylönen 因一次密码嗅探发明 SSH 的全过程，横向对比 SSH vs HTTPS+PAT vs GitHub CLI 三方认证方案的优劣。包含真实的 WSL 踩坑案例分析。

---

## 一、纵向分析：从 Telnet 后门到 GitHub 标配

### 1.1 起源：一次密码嗅探改变了一切

1995 年，芬兰赫尔辛基理工大学。Tatu Ylönen 是一名研究员。某天，他所在的网络遭到密码嗅探攻击——攻击者在路由器上装了嗅探器，轻松截获了大量明文传输的用户名和密码，其中包括多个 root 账号。

当时主流的远程登录协议（Telnet、rlogin、FTP）全部明文传输密码。Ylönen 决定自己写一个替代品。

1995 年 7 月，他发布了 SSH（Secure Shell）第一个版本。最早的 SSH-1 协议使用 RSA 公钥加密，支持安全远程登录、文件传输和端口转发。到 1995 年底，SSH 已获得 50 个国家 20000 名用户。

> **关键决策：** Ylönen 选择「个人免费、企业收费」模式，为后来的开源分裂埋下伏笔。

### 1.2 分裂与重生：OpenSSH 的诞生

1999 年，Ylönen 的公司发布了 SSH-2 协议，但采用了更严格的许可证。这激怒了开源社区。

OpenBSD 创始人 Theo de Raadt 带领团队，基于 SSH-1 最后一个自由版本（1.2.12），从头清理代码、修复漏洞、添加 SSH-2 支持，于 1999 年 10 月发布了 OpenSSH 1.2.2。

这是一个经典的「自由软件复仇故事」：商业公司收紧许可 → 开源社区 fork → 社区版本碾压原版。今天，全球超过 **90%** 的 SSH 服务器运行的是 OpenSSH。

### 1.3 GitHub 与 SSH 的相遇

2008 年 GitHub 上线，从一开始就把 SSH 公钥变成用户身份的「护照」：

| 年份 | 里程碑 |
|------|--------|
| 2008 | 支持上传 SSH 公钥，用 SSH 协议 push/pull |
| 2010 | 成为标准配置，「生成 SSH Key」成为入门第一步 |
| 2013 | 推出 deploy keys，细粒度权限管理 |
| 2019 | 开始推 HTTPS + PAT 替代方案 |
| **2021.08** | **停止接受密码认证**——数百万人被迫学 SSH |

### 1.4 密码出局：2021 年的身份认证大地震

2021 年 8 月 13 日，GitHub 宣布所有 Git 操作不再接受密码认证。背景是账号被盗事件频发。

替代方案只有两个：
1. **SSH**：生成密钥对，公钥上传，私钥本地保管
2. **HTTPS + PAT**：用有限权限的 Token 代替密码

教程网站、技术博客、YouTube 在这个月爆发了「如何设置 GitHub SSH」的内容潮。

### 1.5 算法进化：从 RSA 到 Ed25519

| 算法 | 时间 | 密钥长度 | 状态 |
|------|------|----------|------|
| RSA | 1995 | 2048-4096 bit | 逐渐被取代 |
| DSA | 1995 | 1024 bit | 已被弃用 |
| ECDSA | 2009 | 256-521 bit | 存在争议 |
| **Ed25519** | **2014** | **256 bit** | **推荐方案** |

Ed25519 碾压级优势：密钥更短、签名更快（毫秒级 vs 秒级）、安全性更高（抗侧信道攻击）。

## 二、横向分析：SSH vs HTTPS+PAT vs GitHub CLI

### 2.1 技术路线对比

| 维度 | SSH | HTTPS + PAT | GitHub CLI |
|------|-----|-------------|------------|
| 认证原理 | 非对称密钥 | 对称 Token | OAuth 2.0 |
| 首次设置 | 密钥生成 → 上传 | 网页生成 Token | gh auth login |
| 网络端口 | 22（常被封） | 443（不会被封） | 443 |
| 密钥管理 | 本地文件+密码保护 | 明文存 credential | gh 自动管理 |
| 安全性 | ★★★★★ | ★★★ | ★★★★ |
| 易用性 | ★★★ | ★★★★ | ★★★★★ |

### 2.2 用户真实口碑

**SSH 派：** 老开发者最爱。但新手普遍反馈「第一次配 SSH 很痛苦」「permission denied 什么意思」。

**HTTPS+PAT 派：** 企业内网开发者，被动选择（防火墙封 22 端口）。痛点：「Token 过期了」「忘了存在哪」。

**GitHub CLI 派：** 重度 GitHub 用户。好评最多，但「又多了一个要装的 CLI」「API 变动太快」。

> 🔥 **真实洞察：** 绝大多数人不是「选」了哪种方式，而是跟着搜索引擎第一篇教程走的。「GitHub SSH 设置」每月数百万搜索量——人们不是在选择，而是在照做。

### 2.3 生态位分析

三种方式形成金字塔：
- **底层 HTTPS**：覆盖面最广
- **中层 SSH**：安全性最高
- **顶层 GitHub CLI**：体验最好

三者不是竞争关系，而是不同场景的最优解。

### 2.4 潜在竞争者

- **Passkeys / WebAuthn**：2023 年 GitHub 支持 Web 登录。未来可能扩展到 Git 操作
- **Sigstore / 无密钥签名**：OIDC 身份认证+透明日志，可能让 SSH 密钥彻底退出
- **零信任架构**：每次操作都验证，而非一次密钥永久有效

### 2.5 真实踩坑案例

笔者在 WSL 环境用 HTTPS+PAT 推送时连踩两坑：

1. **DNS 污染**：WSL 把 github.com 解析到 127.0.0.1 → 改 /etc/resolv.conf 到 223.5.5.5 解决
2. **TLS 握手失败**：GnuTLS recv error (-110) → 最终改用 SSH 解决

> **教训：** 国内 WSL 环境，HTTPS 稳定性远不如 SSH。SSH 优先不是因为理论上更安全，而是实践中更可靠。

## 三、横纵交汇：SSH 的未来

### 3.1 SSH 不会消失，但会「退居幕后」

就像 TCP/IP，SSH 正从「用户需要了解的东西」变成「系统自动处理的东西」。未来的开发者可能不会手动运行 ssh-keygen，但 SSH 协议本身仍将是代码传输的底层支柱。

### 3.2 Ed25519 正在成为新默认

RSA 4096 时代落幕。Ed25519 正成为 GitHub、GitLab、Bitbucket 三家共同推荐的默认算法。

### 3.3 「无密钥」是终极目标

Sigstore、OIDC、Passkeys 指向同一个未来：开发者不需要管理任何长期密钥。但这条路至少还要 3-5 年。

### 3.4 实践建议

1. ✅ 用 Ed25519：`ssh-keygen -t ed25519`，不要再 RSA
2. ✅ 给密钥加 passphrase，加到 ssh-agent
3. ✅ 每台机器独立密钥，不复制私钥
4. ✅ 国内 WSL 环境 SSH 优先
5. ✅ 学 gh CLI 作为备选
